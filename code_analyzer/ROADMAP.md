# Code Analyzer 進化の方針（v1.2 → 将来）

v1.2時点の到達点を踏まえ，次期開発の方向性を整理する．前提となる利用像は「大学研究者が個人で，研究コードや調査対象OSSを日常的に解析する」であり，この前提が変わる場合（研究室での共同利用，論文のための大規模実証研究など）は優先順位も変わる．その分岐も含めて示す．

## 現在地の評価

v1.2は「1回の解析を深く見る」ツールとしては一巡した．4言語の統一パース，依存・メトリクス・中心性，データフロー近似つき影響範囲，ソース連動UI，GUIまで揃い，184テストで品質を担保している．一方で，現在の設計には次の3つの構造的な限界がある．

1. **時間軸がない** — 解析は常に「いまのスナップショット」であり，コードの変化・劣化・改善を捉えられない．
2. **解決精度が名前ベース** — シンボル解決は構文と名前の一致に依存し，型情報を使わないため，同名メソッドや動的ディスパッチで偽陽性/偽陰性が残る（現状は信頼度タグで明示する設計で緩和）．
3. **毎回フル解析** — 中規模までは高速（Flask 83ファイルで約1秒）だが，数千ファイル級や繰り返し利用では待ち時間とレポート肥大が効いてくる．

進化の軸はこの3つの限界にそれぞれ対応させるのが素直であり，加えて研究者固有の第4の軸（研究基盤化）がある．

## 軸1: 時間軸の導入（推奨: 次期 v1.3 の中核）

**git履歴との統合**により，スナップショット解析ツールを「コードの変化を観測する装置」に進化させる．

- **ホットスポット分析**: コミット履歴から得た変更頻度（churn）と既存の複雑度メトリクスを掛け合わせ，「複雑かつ頻繁に触られる」危険領域をツリーマップ・散布図で可視化する．実装コストが低い割に意思決定への効用が最も大きい，古典的だが強力な指標．
- **diff影響解析**: `code-analyzer impact --diff HEAD~1..HEAD`（またはワーキングツリーの未コミット差分）で，変更された行範囲を自動抽出し既存の影響範囲エンジンに投入する．「この行を変えたら」から「この変更で」へ．現行の `FILE:START-END` 入力は差分の各ハンクにそのまま対応づくため，エンジン改修はほぼ不要で入口の追加だけで済む．
- **スナップショット比較ビュー**: 2つの `analysis.json` を突き合わせ，メトリクスの増減・依存エッジの追加/削除・循環依存の発生/解消を差分レポートとして表示する．スキーマが安定・決定的である現在の設計はこの用途を最初から支えている．

この軸を推す理由: 既存資産（影響エンジン・決定的JSON・レポート基盤）をほぼそのまま再利用でき，研究・実務の双方で「いつ悪くなったのか」「この変更は安全か」という最頻出の問いに直接答えるため，投資対効果が最大となる．

## 軸2: 解決精度の深化（中期 v1.4〜）

- **Python先行の型認識解決**: 言語ごとに一律の精度向上を狙うのではなく，利用頻度が最も高いPythonに絞って段階的に導入する．具体的には (a) 代入・戻り値の型ヒントからのレシーバ型推定，(b) クラス階層を用いたメソッド解決候補の絞り込み．曖昧エッジの候補数を減らすだけでも影響解析の偽陽性は目に見えて減る．
- **C/C++のcompile_commands本格活用**: 現在はインクルードパスのみの反映だが，マクロ定義を考慮した条件付き解析（少なくとも `#ifdef` の分岐両側を「条件付きエッジ」として保持）へ拡張する．
- **精度の自己測定**: 解決率（certain/inferred/unresolvedの比率）をメトリクスとしてレポートに常設し，バージョン間で精度改善を定量追跡できるようにする．研究ツールとして「自分の測定誤差を自分で報告する」性質は信頼性の核になる．

留意点: この軸は工数対効果の逓減が速い．完全な型推論は言語処理系の再実装に近づくため，「候補集合を狭める」以上を狙わないという線引きを守るべき．

## 軸3: スケールと応答性（必要になった時点で）

- **増分解析の本格化**: 現在の簡易キャッシュを，ファイルハッシュ単位でシンボル・エッジ・メトリクスを差し替える真の増分更新に拡張する．SQLiteを中間ストアにすると比較ビュー（軸1）とも相性がよい．
- **watchモード**: GUIサーバは常駐しているため，ファイル変更を監視して自動再解析・自動リロードする拡張は自然な延長線上にある．
- **レポートの遅延読み込み**: 現在の単一HTML方式は5千ファイル規模で限界が来る．その際は「サマリHTML＋データを別ファイル群に分割し `--serve` で閲覧」というハイブリッドに移行する．単一HTMLの配布容易性は小中規模向けに残す．

この軸は「困ってから」で遅くない．現在の性能で足りている限り優先度は低い．

## 軸4: 研究基盤化（利用像が広がる場合）

- **バッチ解析**: リポジトリ一覧（CSV/対象ディレクトリ群）を与えて一括解析し，横断メトリクスを1つのデータセット（CSV/Parquet）に集約する `code-analyzer batch`．実証研究のデータ収集を1コマンド化する．
- **再現性パッケージ**: 解析設定・ツールバージョン・対象リビジョンを固めた「解析マニフェスト」を出力し，論文の付録としてそのまま添付できる形式にする．
- **メソッド文書**: 各メトリクスの算出定義は既にレポート同梱だが，これを引用可能な形（DOI付きの技術文書，あるいはCITATION.cff）に整える．

## 推奨ロードマップ

| バージョン | 中核 | 内容 |
|---|---|---|
| v1.3 | 時間軸 | diff影響解析（`--diff`），ホットスポット（churn×複雑度），スナップショット比較ビュー |
| v1.4 | 精度 | Python型認識解決（候補絞り込み），解決率メトリクスの常設，C/C++マクロ条件エッジ |
| v1.5 | 応答性 | SQLite増分解析，GUI watchモード，（必要なら）レポート分割配信 |
| v2.0 | 研究基盤 | バッチ解析＋横断データセット出力，解析マニフェスト，比較の統計サマリ |

## 意図的にやらないこと

方針として見送る項目も明示しておく．**VS Code拡張/LSP化**は魅力的だが，エディタ統合は保守負担が大きく，単一HTMLレポートという本ツールの独自価値（配布・共有・オフライン）と競合するため，diff影響解析のCLI/GUIで代替できる範囲では着手しない．**言語追加**（Go/Rust等）はtree-sitterの構造上コストは低いものの，1言語増えるごとに解決器・メトリクス・テストの保守が線形に増えるため，実際に解析したい対象が現れてから行う．**チーム向けサーバ常駐・認証・DB共有**は個人ツールとしての軽さを失うため，研究室共同利用の具体的な需要が生じるまで保留する．

## 次の一歩

v1.3の3機能のうち，最初に着手すべきは **diff影響解析** である．理由は，(1) 既存エンジンの入口追加のみで実装リスクが最小，(2) 「コードを変更する直前・直後」という利用頻度が最も高い瞬間にツールを組み込める，(3) ホットスポットと比較ビューが必要とするgit連携の基盤（リビジョン指定・差分抽出）を先に整備できるため．
